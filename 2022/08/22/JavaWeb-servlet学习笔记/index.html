JavaWeb-servlet学习笔记

发表于2022-08-22|更新于2022-08-24
|阅读量:

servlet快速入门

1.创建web项目,导入servlet依赖坐标,标签必须标记为provided

2.定义一个类,实现servlet接口,并重写接口中所有方法,并在service方法中输入一句话.

3.配置:在类上用@WebServlet注解,配置该servlet的访问路径

4.启动tomcat,访问该servlet

在使用servlet时,创建了servlet的类,但是没有实例化,也没有调用service方法,方法依旧生效.

其实是在项目运行的时候,tomcat将servlet类实例化并对方法进行了调用.

servlet生命周期

1.加载和实例化

默认情况下,当servlet第一次被访问时,由容器创建servlet对象

2.初始化

在servlet实例化之后,容器将调用servlet的init()方法初始化这个对象,完成一些如加载配置文件,创建链接等初始化工作.该方法只调用一次.

3.请求处理

每次请求servlet时,servlet容器都会调用servlet的service()方法对请求进行处理.

4.服务终止

当需要内存或者容器关闭时,容器就会调用Servlet实例的destroy()方法完成资源的释放,在destory()方法完成后,容器会释放这个servlet实例,该实例随后会被Java的垃圾回收器回收

  • 由于servlet中除了servce()以外的四个方法并不常用,写servlet类时需要额外重写四个方法十分的繁琐,因此会使用自定义的servlet类继承HttpServlet类

    servlet urlPattern配置

  • servlet想要被访问,必须配置其urlPattern

1.一个servlet可以配置多个urlPattern

@WebServlet(urlPatterns = {"/demo3","/demo4"})

urlPattern的匹配规则

精确匹配

@WebServlet(urlPatterns = "/user/select")

目录匹配

@WebServlet("/user/*")

扩展名匹配

@WebServlet("*.do")

不可以/开头

任意匹配

@WebServlet("/")

@WebServlet("/*")

当项目中的servlet配置了"/",会覆盖掉tomcat中的defaultservlet,当其他的url-pattern都匹配不上时会走这个servlet

在项目中覆盖掉默认servlet时,会使项目无法访问静态资源,非必要不要配置

当项目中配置了"/*"时,意味着匹配任意访问路径

匹配优先级

精确路径 > 目录路径 > 扩展名路径 > /* > /

XML配置servlet

  • servlet从3.0开始支持使用注解配置,3.0版本前只支持xml配置servlet方式

    步骤:

    • 编写Servlet类
    • 在web.xml中配置该Servlet

    Request和Response

    Request请求转发

    想要在"/Demo1"的servlet中同时触发"/Demo2"的Servlet响应,只需要添加

    request.getRequestDispatcher("/Demo2").forward(request, response);

    请求转发资源间共享数据:使用Request对象

  • void setAttribute(String name, Object o);存储数据到request域中
  • Object getAttribute(String name);根据key,获取值
  • void removeAttribute(String name);根据key,删除键值对

    Response响应数据

    设置响应数据

  • void setStatus(int sc)设置响应状态码
  • void setHeader(String name, String value)设置响应头键值对
  • PrintWriter getWriter()获取字符输出流

    ServletOutputStream getOutputStream()获取字节输出流

    Response重定向

    在被请求的资源无法处理请求时,可以使用重定向告诉浏览器另外的资源地址,让浏览器重新对新的地址进行请求 tomcat还提供了一种简化的写法 重定向特点:
  • 浏览器地址栏路径发生变化
  • 可以重定向到任意位置的资源(服务器内部,外部均可)
  • 两次请求,不能在多个资源使用request共享数据

    Response设置响应体

    在响应后会将字符串aa输出在浏览器中,如果添加的响应体是<h1>aaa</h1>,浏览器并不会将文本转为标题一格式,因为此时的浏览器是将此字符串当作纯文本解析

    要想将文本以html格式被解析,要添加response.setHeader("content-type", "text/html");

但是在修改了文本类型后,依旧存在一个问题,tomcat在8版本默认编码不是utf-8,响应的数据在传输后中文解码会变成乱码

解决方法:

响应字节流数据

但是数据流的拷贝有些繁琐,可以使用apache提供的一个工具commons-io来简化操作

使用此工具前需要导入工具依赖

使用工具的copy方法即可完成流的复制
